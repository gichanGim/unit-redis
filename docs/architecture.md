# 아키텍처 — 단위 레디스 (unit-redis)

> Phase 1 시작 시점(이슈 #1)에 확정한 구조와 그 이유, 그리고 미래 Phase에서 재검토할 체크포인트를 기록한다.

## 한 줄 요약

**경계(코어↔외부)에는 헥사고날, 내부는 평범한 레이어드, 교체가 필요한 지점마다 DIP.**
헥사고날과 DIP는 같은 도구(인터페이스)를 쓰지만 목적이 다르다 — 헥사고날은 **격리**, DIP는 **교체**.

## 구조도

```
redis-cli
 │  RESP2 over TCP (P2)
 ▼
서버 레이어 — 어댑터 (P2)
 ├─ TCP 서버 (가상 스레드)
 ├─ RESP2 파서
 └─ 커맨드 디스패처
 │
 ▼
코어 엔진 (P1 · 네트워크를 모른다)
 │
 ├─ Storage 인터페이스 ── ★ 인바운드 포트
 │   * 단위 · 오라클 테스트는 서버를 거치지 않고 여기로 직접 들어온다 (P1~)
 │    ▲ implements
 ├─ InMemoryStorage (P1)
 │   * P4에서 교체: 전역 락 / CHM / 단일 스레드+큐
 │    │ uses
 │    ▼
 ├─ Dict 인터페이스 ── 내부 DIP (포트 아님)
 │    ▲ implements
 └─ ChainingDict (P1 · 직접 구현 해시 테이블)
```

의존(화살표)은 항상 바깥→안, 위→아래로만 향한다. 역방향이 하나라도 생기면 설계가 깨진 것.

## 패키지 구조 (Phase 1)

```
src/main/java/unitredis/
├── Main.java                      # 빈 껍데기 (Phase 2에서 서버 부트스트랩)
└── core/
    ├── Storage.java               # ★ 명령 시맨틱 층 인터페이스 — Phase 4 교체 지점
    ├── InMemoryStorage.java       # 기본 구현 (처음엔 HashMap 스텁 → Dict로 교체)
    ├── RedisException.java        # 에러 모델 (Phase 2에서 RESP `-ERR ...`로 변환)
    └── dict/
        ├── Dict.java              # 해시 테이블 인터페이스 (put/get/remove/size)
        └── ChainingDict.java      # 직접 구현 (v1 chaining → v2 리사이즈)

src/test/java/unitredis/core/      # main 구조를 미러링
```

## 두 개의 층, 두 개의 인터페이스

| | `Storage` | `Dict` |
| --- | --- | --- |
| 역할 | 명령 시맨틱 (Redis 문서를 아는 층) | 순수 자료구조 (Redis를 모르는 층) |
| 성격 | **헥사고날 인바운드 포트** — 바깥 계약 | **내부 DIP** — 사적 인터페이스 |
| 시그니처 변경 비용 | 비쌈. 신중하게, 독립 커밋으로 | 쌈. 부담 없이 리팩토링 가능 |
| 교체 시나리오 | P4 동시성 모델 3종 (전역 락 래퍼 / CHM 기반 / 단일 스레드+커맨드 큐) | P1 v1→v2, P4 CHM 비교 |

역할 분리 기준: **"이 코드가 Redis 문서를 알아야 하는가?"** — 알아야 하면 Storage 구현(예: INCR 오버플로우 에러), 몰라도 되면 dict(예: 버킷 리사이즈).

## 의존 방향 규칙 (위반 = 리뷰에서 지적할 것)

1. `dict`는 아무것도 import하지 않는다 — 표준 라이브러리만
2. `core`는 `java.net`, `java.nio` 계열을 절대 import하지 않는다 (모든 Phase에서)
3. Phase 2의 `unitredis.resp`, `unitredis.server`는 `core`를 의존하고, 역방향 금지
4. 서버 레이어의 타입(RESP 응답 문자열, 소켓 예외 등)이 코어 시그니처에 새어 들어오면 안 됨

## 헥사고날 경계의 위치

- **인바운드(driving)**: redis-cli → 서버 어댑터(P2), 그리고 테스트(P1~). 둘 다 `Storage` 포트로 합류.
  Phase 1의 어댑터는 단위 테스트 그 자체다. Phase 2에서 TCP 어댑터가 추가돼도 코어는 한 줄도 안 바뀌는 게 목표.
- **아웃바운드(driven)**: 현재 없음. **Phase 7 영속화가 생기면 두 번째 경계가 된다** — 코어는 "스냅샷을 기록하라"는 포트만 알고, 파일 I/O 어댑터는 바깥에 둔다.

## 미리 만들지 않는 것 (YAGNI — 자리만 인지)

- **커맨드 디스패처 / Command 패턴** — Phase 2에서 명령 이름 → 코어 라우팅이 필요해질 때 도입
- **값 타입 래퍼(RedisObject류)** — Phase 5 컬렉션 때
- **TTL 메타데이터** — Phase 3에서 Storage 인터페이스 확장 (만료 시각은 절대 Unix ms)

## 알려진 약점과 재검토 체크포인트

아키텍처 점검(2026-07-22) 결과. 약점은 없애는 게 아니라 터질 위치·시점을 알고 있는 것이 목표.

| 체크포인트 | 시점 | 내용 |
| --- | --- | --- |
| 동기 인터페이스 가정 검증 | **Phase 4 진입 시** | `Storage`는 동기 인터페이스인데 단일 스레드+큐 모델은 본질이 비동기. 호출자가 Future를 블로킹 대기하는 래퍼로 성립 가능(가상 스레드라 대기가 쌈 — JEP 444). 이 가정은 보장이 아니라 검증 대상. |
| 인터페이스 분리 결정 | **Phase 5 진입 전** | 컬렉션 명령이 들어오면 `Storage` 단일 인터페이스가 비대해짐. P2에서 디스패처가 생기면 서버의 창구는 디스패처가 되므로, Storage를 타입군별로 쪼갤 자유가 생김 (실제 Redis의 `server.c` 명령 테이블 + `t_string.c`/`t_list.c` 분리와 같은 구조). |
| 블로킹 명령 대응 | **Phase 8 (선택)** | BLPOP은 동기 요청-응답 모양을 벗어나고 MULTI/EXEC는 명령의 객체 표현이 필요. P2 디스패처의 연장선으로 수용 가능 — 구조 전복은 아님. |

## 기각한 대안 (다시 제안하지 말 것)

- **인터페이스 없이 단일 패키지, 필요할 때 리팩토링**: Phase 4의 "구현 교체 비교" 자체가 학습 목표라 교체 지점이 요구사항. 기각.
- **처음부터 커맨드 패턴**: Phase 1에서는 배움 없이 보일러플레이트만 늘어남. Phase 2로 연기.
- **처음부터 단일 스레드 이벤트 루프(실제 Redis 모양)**: 하나를 구조에 박으면 Phase 4의 3모델 비교 실험이 불가. 기각.
- **풀 헥사고날(포트 세분화 · 경계마다 DTO)**: 1인 학습 프로젝트에서 배움 없는 의식(ceremony). 경계에만 선별 적용.
